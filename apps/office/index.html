            📶 Automated Client Notifications (off by default)
            When enabled, DeepFlow sends a small event (job status changed, certificate ready) to a webhook URL you configure. That webhook is where you connect a free automation tool — n8n (self-hosted, free), Zapier or Make (free tiers) — which actually sends the WhatsApp message, email, or SMS. DeepFlow itself doesn't send messages directly; no WhatsApp Business API or paid SMS account is required to turn this on, only to make it actually deliver something once it's on.
            🔔 Real Push Notifications (off by default, free)
            A real phone notification — no third-party service, no account, no cost — using each browser's own free push service. Clients turn this on themselves from a "Get notified on your phone" prompt in their portal; there's nothing for them to install. This needs a small piece deployed on your Supabase project first (a one-time technical setup — see PHASE6_PUSH_NOTIFICATIONS_SQL.md and PHASE6B_PUSH_EDGE_FUNCTION.md). Uses the same "Notify on job status change / certificate ready" toggles above.
            📲 Next-Tenant ETA (off by default, free)
            When an engineer marks a job Completed, this looks for their next Pending job that day and, if someone needs to let them in (access is set to "Tenant Home" or "Landlord Present" with a phone number, not a key safe code), fires a webhook event so your connected automation can text them the engineer is on the way. Uses the same webhook URL configured above — no separate setup. This is a heads-up, not a precise timed ETA (no live traffic/distance data yet), so the message should read as "on their way" rather than a specific arrival time.

            ⏰ Task 27 — Scheduled Cert Expiry Reminders (pg_cron)
            Automatic daily WhatsApp reminder for expiring certificates
            Run this SQL once in Supabase. It creates:
            1. A cert_reminder_log table to prevent duplicate sends
            2. A send_cert_reminders() function that finds certs expiring within the reminder window and logs them
            3. A pg_cron job that runs daily at 9am UTC
            Note: pg_cron must be enabled in your Supabase project under Database → Extensions. WhatsApp is opened via a webhook — replace WEBHOOK_URL with your own (e.g. a Make/Zapier webhook that calls the WA Business API).
            -- ══ STEP 1: Enable pg_cron extension (do this in Extensions tab first) ══ -- CREATE EXTENSION IF NOT EXISTS pg_cron; -- ══ STEP 2: Reminder log table ══ CREATE TABLE IF NOT EXISTS cert_reminder_log ( id text PRIMARY KEY DEFAULT gen_random_uuid()::text, cert_id text NOT NULL, sent_at timestamptz NOT NULL DEFAULT now(), days_before int, method text DEFAULT 'whatsapp' ); CREATE INDEX IF NOT EXISTS idx_cert_reminder_log_cert ON cert_reminder_log(cert_id, sent_at); -- ══ STEP 3: Function that finds certs needing reminders ══ CREATE OR REPLACE FUNCTION send_cert_reminders() RETURNS TABLE(cert_id text, address text, landlord text, cert_type text, expiry_date date, days_left int, phone text) LANGUAGE plpgsql SECURITY DEFINER AS $$ DECLARE warn_days int[] := ARRAY[60, 30, 14, 7, 1]; -- send reminders at these thresholds d int; BEGIN FOREACH d IN ARRAY warn_days LOOP FOR cert_id, address, landlord, cert_type, expiry_date, days_left, phone IN SELECT c.id, c.address, c.landlord, c.type, c.expirydate::date, (c.expirydate::date - CURRENT_DATE) as days_left, COALESCE(p.wa, p.phone, '') as phone FROM certs c LEFT JOIN persons p ON lower(p.name) = lower(c.landlord) WHERE c.expirydate IS NOT NULL AND c.noexpiry IS NOT TRUE AND (c.expirydate::date - CURRENT_DATE) = d AND NOT EXISTS ( SELECT 1 FROM cert_reminder_log rl WHERE rl.cert_id = c.id AND rl.days_before = d AND rl.sent_at > now() - interval '2 days' ) LOOP -- Log the reminder (app/webhook reads this table to send WA messages) INSERT INTO cert_reminder_log(cert_id, days_before, sent_at) VALUES (cert_id, d, now()) ON CONFLICT DO NOTHING; -- Return the row so the caller can send the actual message RETURN NEXT; END LOOP; END LOOP; END; $$; -- ══ STEP 4: Schedule daily at 9am UTC ══ SELECT cron.schedule( 'deepflow-cert-reminders', '0 9 * * *', $$ SELECT send_cert_reminders(); $$ ); -- ══ CHECK: view scheduled jobs ══ -- SELECT * FROM cron.job WHERE jobname = 'deepflow-cert-reminders'; -- ══ UNSCHEDULE if needed ══ -- SELECT cron.unschedule('deepflow-cert-reminders');
            How to connect this to WhatsApp:
            The function logs pending reminders to cert_reminder_log. To actually send the messages, set up a Make / Zapier / n8n workflow that:
            1. Runs every morning, queries cert_reminder_log for today's entries
            2. For each row, looks up the phone number and sends a WhatsApp message
            3. Or use the manual bulk-reminder button in Certificates → Reminders for immediate sends

            🔐 Fix 1 — Can't Create Staff or Engineers?
            If creating office staff or engineers fails with an RLS error, run this SQL once in Supabase.
            -- Fix users table RLS policy (run if Create Account fails) DROP POLICY IF EXISTS "auth_users_write" ON users; DROP POLICY IF EXISTS "auth_users_read" ON users; CREATE POLICY "users_read" ON users FOR SELECT TO authenticated USING (true); CREATE POLICY "users_insert" ON users FOR INSERT TO authenticated WITH CHECK (true); CREATE POLICY "users_update" ON users FOR UPDATE TO authenticated USING (EXISTS (SELECT 1 FROM users u WHERE u.auth_id = auth.uid() AND u.role = 'admin' AND u.active = true)) WITH CHECK (true); CREATE POLICY "users_delete" ON users FOR DELETE TO authenticated USING (EXISTS (SELECT 1 FROM users u WHERE u.auth_id = auth.uid() AND u.role = 'admin' AND u.active = true));
            🔄 Enable "Sync from Supabase" button
            Run this SQL once to enable the Sync from Supabase button. It creates a safe function that lets DeepFlow read your Supabase Auth users list.
            -- Run ONCE — enables Sync from Supabase button in Settings → Users CREATE OR REPLACE FUNCTION get_auth_users() RETURNS TABLE(id uuid, email text, created_at timestamptz) LANGUAGE plpgsql SECURITY DEFINER SET search_path = auth, public AS $$ BEGIN RETURN QUERY SELECT u.id, u.email::text, u.created_at FROM auth.users u ORDER BY u.created_at DESC; END; $$;
            ✅ Fix 2 — Staff can't log in (email confirmation required)?
            Run this SQL once. It creates a function that makes new staff accounts instantly active — no email confirmation ever needed. After this, every new staff member you create can log in immediately.
            -- ══ Create confirmed user function (run ONCE — permanent fix) ══ -- After this, all new office staff accounts work immediately with no email confirmation. CREATE OR REPLACE FUNCTION create_confirmed_user(p_email text, p_password text) RETURNS uuid LANGUAGE plpgsql SECURITY DEFINER SET search_path = auth, public AS $$ DECLARE v_user_id uuid; v_encrypted_pw text; BEGIN v_user_id := gen_random_uuid(); v_encrypted_pw := crypt(p_password, gen_salt('bf')); INSERT INTO auth.users ( instance_id, id, aud, role, email, encrypted_password, email_confirmed_at, recovery_sent_at, last_sign_in_at, raw_app_meta_data, raw_user_meta_data, created_at, updated_at, confirmation_token, email_change, email_change_token_new, recovery_token ) VALUES ( '00000000-0000-0000-0000-000000000000', v_user_id, 'authenticated', 'authenticated', p_email, v_encrypted_pw, now(), -- email_confirmed_at = NOW = pre-confirmed, no email needed now(), now(), '{"provider":"email","providers":["email"]}', '{}', now(), now(), '', '', '', '' ); INSERT INTO auth.identities ( id, user_id, provider_id, identity_data, provider, last_sign_in_at, created_at, updated_at ) VALUES ( gen_random_uuid(), v_user_id, p_email, json_build_object('sub', v_user_id::text, 'email', p_email), 'email', now(), now(), now() ); RETURN v_user_id; END; $$;
            After running this SQL, go to Settings → Users & Access and use the "🔑 Create Login" button on any existing "⚠ No login yet" users to fix them instantly too.
