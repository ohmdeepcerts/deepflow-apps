# 11 — APIs

DeepFlow has **no custom API of its own** — every "API call" the three apps make is a direct call to one of Supabase's own auto-generated services. This document catalogues every kind of call made, with real example shapes (using synthetic, non-real data throughout). All examples use the project's real base URL structure with a placeholder key, since the actual anon key is already public in the apps' own source (see [15_Security.md](15_Security.md) for why that's expected, and what the real risk actually is).

**Base URLs:**
- REST (database): `https://dzqyqpuhxdrrpipbehpk.supabase.co/rest/v1/`
- Auth: `https://dzqyqpuhxdrrpipbehpk.supabase.co/auth/v1/`
- Storage: `https://dzqyqpuhxdrrpipbehpk.supabase.co/storage/v1/`

## 1. Database API (PostgREST) — the pattern used for almost everything

**Authentication:** every request carries two headers — `apikey: <anon key>` (identifies the project) and `Authorization: Bearer <token>` (either the anon key itself for public/no-login contexts, or the logged-in user's session token for authenticated contexts). **Validation:** performed entirely by the requesting app before sending, and separately by Postgres's own column types/constraints on arrival — there is no intermediate validation layer. **Errors:** standard HTTP status codes plus a JSON body (`{code, message, details, hint}`) generated by PostgREST/Postgres directly.

### 1.1 Read (GET) — example: fetch today's jobs for one engineer (Employee App)

**Request**
```
GET /rest/v1/jobs?date=eq.2026-07-15&engineer=ilike.J.%20Smith&order=created.asc&select=*
apikey: <anon key>
Authorization: Bearer <engineer's session token>
```

**Response — 200 OK**
```json
[
  {
    "id": "job-eng-1773000000-ab12",
    "jobnum": "JOB-1004",
    "address": "12 Example Street, Sample Town",
    "date": "2026-07-15",
    "timeslot": "09:00-11:00",
    "engineer": "J. Smith",
    "status": "Pending",
    "price": 150,
    "created": 1773000000000
  }
]
```

**Response — 401 Unauthorized** (expired/invalid token)
```json
{ "code": "PGRST301", "message": "JWT expired" }
```

### 1.2 Create (POST, upsert) — example: save a new job (Office App)

**Request**
```
POST /rest/v1/jobs
apikey: <anon key>
Authorization: Bearer <office user's session token>
Content-Type: application/json
Prefer: resolution=merge-duplicates,return=minimal

{ "id": "a1b2c3d4-...", "jobnum": "JOB-1005", "address": "5 Example Ave", "status": "Pending", "created": 1773000100000 }
```
**Response — 201 Created** (empty body, because `return=minimal` was requested) **Response — 401**, if RLS rejects the write:
```json
{ "code": "42501", "message": "new row violates row-level security policy for table \"jobs\"" }
```
*(Live-tested for other tables — see [15_Security.md](15_Security.md) Section 3.2 — anonymous inserts are rejected on several tables; this exact status/body is what that rejection looks like.)*

### 1.3 Targeted Update (PATCH) — example: change a job's status only

**Request**
```
PATCH /rest/v1/jobs?id=eq.a1b2c3d4-...
apikey: <anon key>
Authorization: Bearer <token>
Content-Type: application/json
Prefer: return=minimal

{ "status": "Completed", "modified": 1773000200000 }
```
**Why this matters architecturally:** this narrower, single-field PATCH (rather than sending the whole job object) is the app's main conflict-mitigation technique — see [10_Synchronization.md](10_Synchronization.md) Section 4.

### 1.4 Delete (DELETE) — example: remove a job

**Request**
```
DELETE /rest/v1/jobs?id=eq.a1b2c3d4-...
apikey: <anon key>
Authorization: Bearer <token>
Prefer: return=minimal
```
**Response — 200 OK**, body `[]`. **Important caveat, confirmed by direct testing:** a delete filter that matches **zero** rows also returns `200 OK` with `[]` — this looks identical to "the delete succeeded" whether or not a real row was actually deleted, which is why [15_Security.md](15_Security.md) Section 3.3 could not use this method to conclusively test delete permissions against real data.

### 1.5 Counting rows without downloading them

**Request**
```
GET /rest/v1/jobs?select=id&limit=0
Prefer: count=exact
```
**Response headers include:** `Content-Range: */3` (three total rows) — used throughout this documentation set to report table sizes without exposing any actual row content.

## 2. Auth API

Used by the Office and Employee apps only, via the `@supabase/supabase-js` library rather than hand-written `fetch()` calls (the one place these two apps use the official SDK for more than just Realtime).

### 2.1 Sign in
**Call:** `supabase.auth.signInWithPassword({ email, password })`. **Underlying request:** `POST /auth/v1/token?grant_type=password`. **Response (success):** a session object containing `access_token` (the JWT used for every subsequent PostgREST/Storage call), `refresh_token`, and a `user` object with the Auth account's `id`/`email`. **Response (failure):** `400` with a message like `"Invalid login credentials"`.

### 2.2 Sign out
**Call:** `supabase.auth.signOut()`. Invalidates the session server-side and clears the library's local copy.

### 2.3 Password reset
**Call:** `supabase.auth.resetPasswordForEmail(email)`. Triggers Supabase's own transactional email — no custom email service is involved anywhere in this project.

### 2.4 Get current session (used to build the `Authorization` header for every REST call)
**Call:** `supabase.auth.getSession()`. Returns the current `access_token` if a session exists, or `null`. Both apps' `_getJWT()` helper falls back to the raw anon key if no session is present, so unauthenticated reads still work.

## 3. Storage API

### 3.1 Upload a file
**Request**
```
POST /storage/v1/object/deepflow/jobs/a1b2c3d4-.../1773000300-x7q2.jpg
apikey: <anon key>
Authorization: Bearer <token>
Content-Type: image/jpeg
x-upsert: true

<raw file bytes>
```
**Response — 200 OK**
```json
{ "Key": "deepflow/jobs/a1b2c3d4-.../1773000300-x7q2.jpg", "Id": "<storage object uuid>" }
```
**Security note:** live-tested directly — this succeeds with only the public anon key, no login required at all (see [15_Security.md](15_Security.md) Section 5).

### 3.2 List files in a folder
**Request**
```
POST /storage/v1/object/list/deepflow
Content-Type: application/json
{ "prefix": "jobs/", "limit": 100 }
```
**Response:** an array of `{name, id, created_at, updated_at, metadata}` objects — file content is not included, only metadata.

### 3.3 Delete a file
**Request:** `DELETE /storage/v1/object/deepflow/<path>`. **Response — 200 OK:** `{"message":"Successfully deleted"}`.

### 3.4 Public download URL (no API call needed)
```
GET /storage/v1/object/public/deepflow/jobs/a1b2c3d4-.../1773000300-x7q2.jpg
```
Works in a plain `<img src="...">` tag, with no headers or authentication at all — this is how every photo displays throughout all three apps.

## 4. RPC (Remote Procedure Call) API

**Pattern:** `POST /rest/v1/rpc/<function name>`, with a JSON body of named parameters. Full catalogue of every function referenced in the codebase, and its live status: [06_Supabase.md](06_Supabase.md) Section 6.

### Example — `get_auth_users()`
**Request**
```
POST /rest/v1/rpc/get_auth_users
apikey: <anon key>
Authorization: Bearer <anon key>
Content-Type: application/json
{}
```
**Response — 200 OK**
```json
[
  { "id": "uuid-here", "email": "example@company.com", "created_at": "2026-03-17T18:08:34Z" }
]
```
**Security note:** this succeeds with the plain anon key — no login required — which is the single most severe access-control finding in this project. See [15_Security.md](15_Security.md) Section 3.4 for the full explanation and the one-line fix.

### Example — calling a function that doesn't exist
**Request:** `POST /rest/v1/rpc/exec_sql` **Response — 404**
```json
{ "code": "PGRST202", "message": "Could not find the function public.exec_sql without parameters in the schema cache" }
```
This exact response is how this documentation set confirmed, live, which of the admin panel's documented "setup" functions were actually installed — see [07_SQL_Migrations.md](07_SQL_Migrations.md).

## 5. External (Non-Supabase) APIs

Used only by the Employee App's Map tab, all free and requiring no API key:

| API | Used for | Example call |
|---|---|---|
| postcodes.io | UK postcode → coordinates (primary geocoder) | `GET https://api.postcodes.io/postcodes/<postcode>` |
| Nominatim (OpenStreetMap) | Full-address geocoding fallback | `GET https://nominatim.openstreetmap.org/search?q=<address>&format=json` |
| Open-Meteo | Current weather for the dashboard widget | `GET https://api.open-meteo.com/v1/forecast?latitude=..&longitude=..` |
| UK Land Registry (data.gov.uk) | Recent sale-price lookups by postcode | `GET https://landregistry.data.gov.uk/data/ppi/address-search?postcode=..` |

No API keys are embedded for any of these — all confirmed free/keyless services.
