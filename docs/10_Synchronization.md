# 10 — Synchronization

The full, detailed version of this document is [../SYNCHRONIZATION.md](../SYNCHRONIZATION.md) (10 scenario walkthroughs, a complete trigger table, and full failure-recovery mechanics) — this file is the structured summary, organised to match the rest of this documentation set.

## 1. The Governing Fact

**No app ever sends data directly to another app.** Every app writes to the shared Supabase database; every other app either gets pushed that change live (Realtime — one table, one app, see Section 3) or discovers it later by asking the database on its own schedule (polling) or simply loading a screen fresh. "Syncing," in this system, almost always means **polling**, not pushing.

## 2. Who Initiates Each Sync

| Sync | Initiated by |
|---|---|
| A job appearing live on another Office screen | The person who made the change (Realtime push) |
| Employee App's job list refreshing | A 30-second timer, or a manual pull-to-refresh |
| Employee App seeing a new broadcast alert | A 15-second timer |
| Office App falling back to polling | Automatic, only when Realtime is disconnected |
| Client Portal showing anything | The client opening or manually reloading the link |
| A photo appearing in the Office App | Whoever next opens that specific job |
| GPS position updating | The Employee App's device, automatically, whenever its location sensor reports a change |

## 3. When Data Moves

Full table in [../SYNCHRONIZATION.md](../SYNCHRONIZATION.md) Section 3. Key figures: Realtime push is near-instant, for `jobs` only, Office App only. Employee App job list: within 30 seconds. Broadcast alerts: within 15 seconds. Everything else (invoices, certificates, photos, directory records, settings): only when the relevant screen is next opened — there is no notification telling anyone "this changed elsewhere."

## 4. How Conflicts Are Prevented

There is no field-level locking and no server-side merge. What exists: targeted single-field updates for common actions (like a status change) instead of full-form overwrites; an advisory (not blocking) warning if a job you have open was just changed by someone else; and an automatic "move this invoice back to Draft" safeguard if its linked job changes underneath it. What is genuinely **not** prevented: two people editing the *same* field on the *same* record within the same few seconds — whoever saves last simply wins. Full detail: [../SYNCHRONIZATION.md](../SYNCHRONIZATION.md) Section 4.

## 5. How Duplicate Records Are Prevented

Every save uses a database-level "insert or update" (upsert) pattern keyed on `id`, so re-saving the same record is always safe. Human-readable numbers (job/invoice/certificate/request-reference numbers) are generated by scanning existing records and taking the highest number plus one — safe in ordinary single-user-at-a-time use, but not airtight against two near-simultaneous creations (see the worked example in [../SYNCHRONIZATION.md](../SYNCHRONIZATION.md) Scenario 8.10). Full rule set: [13_Business_Rules.md](13_Business_Rules.md) Section 5.

## 6. How Deleted Records Propagate

A deleted job disappears live from the Office App (Realtime) and within 30 seconds from the Employee App (polling). **Nothing cascades automatically** — certificates, invoices, comments, and attachments linked to a deleted job are left as permanent orphans; deleting a directory record breaks that person's Client Portal link instantly but leaves every job that already copied their name onto it untouched. Full detail: [../SYNCHRONIZATION.md](../SYNCHRONIZATION.md) Section 6.

## 7. Offline Behaviour

**There is no offline write queue anywhere in this system.** If a save fails because the device has no connection, the data is simply lost — not queued, not retried automatically. The Office App shows a clear "Offline" indicator and warns before letting a tab close mid-save; the Employee App shows an error toast and expects the user to retry manually once reconnected. Full detail: [../SYNCHRONIZATION.md](../SYNCHRONIZATION.md) Section 7 and Scenarios 8.2–8.3.

## 8. Retry Behaviour

- **Realtime reconnection:** automatic, attempted every 10 seconds until it succeeds.
- **Polling fallback:** an efficient "has anything actually changed?" check every 5 seconds (Office App), rather than blindly re-fetching everything.
- **Failed writes:** not automatically retried anywhere in the system — every retry is a manual user action (re-tapping a button, pulling to refresh).

## 9. Failure Recovery

A visible sync-status badge (Live / Syncing / Offline) tracks every in-flight save across the whole Office App session; a browser-native "are you sure you want to leave" prompt appears if the tab is closed while a save is still pending. Full mechanics: [../SYNCHRONIZATION.md](../SYNCHRONIZATION.md) Section 7.

## 10. Race Conditions

Catalogued with full reasoning in [../AUDIT.md](../AUDIT.md) Section 3: non-atomic numbering generation, a non-atomic "check no invoice exists, then create one" guard, and full-form saves that can silently overwrite a more recent concurrent edit.

## 11. Data Consistency

Given the points above, this system favours **availability and simplicity over strict consistency** — it is entirely plausible, under real concurrent use, for two records to briefly or permanently disagree (a duplicate job number, a job whose price no longer matches its invoice, an invoice total that drifted from what a job's price field says) without any error ever being raised. This is a deliberate architectural trade-off inherent to having no backend transaction layer, not a single bug to be fixed — anyone extending this system should design new features with this reality in mind rather than assuming database-level consistency guarantees that don't exist here.
